1. 상속의 치명적인 단점: 강한 결합도
상속은 코드를 재사용하는 가장 쉬운 방법처럼 보이지만, 부모 클래스와 자식 클래스 사이의 결합도를 과도하게 높인다.
캡슐화 위반: 자식 클래스가 부모 클래스의 내부 구현 세부사항을 알아야 하므로 캡슐화의 원칙이 깨진다.
취약한 기반 클래스 문제: 부모 클래스의 퍼블릭 인터페이스가 아닌 내부 구현만 변경되어도 자식 클래스가 함께 깨질 수 있는 현상을 말한다.
동시 수정의 문제: 부모 클래스를 수정할 때 자식 클래스도 함께 수정해야 하는 상황이 빈번하게 발생하여 유지보수가 어려워진다.

2. 설계 개선 전략: "차이를 메서드로 추출하고 위로 올리기"
정리하신 것처럼, 상속으로 인한 피해를 최소화하려면 추상화에 의존해야 한다.
공통 코드 추출: 중복되는 코드가 있다면 유사한 메서드들을 비교하여 공통점은 부모 클래스로 옮기고, 차이점만 별도의 메서드로 추출한다.
추상 클래스와 추상 메서드: 부모 클래스를 추상 클래스로 만들고, 자식 클래스마다 달라지는 로직은 abstract 메서드로 남겨두어 자식 클래스가 스스로 구현하게 한다.
템플릿 메서드 패턴(Template Method Pattern): 부모 클래스에서 알고리즘의 기본 흐름(공통 로직)을 정의하고, 중간에 필요한 처리를 자식 클래스에 위임하는 이 방식은 상속의 위험을 완화하는 대표적인 디자인 패턴이다.
